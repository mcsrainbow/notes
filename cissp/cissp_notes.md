```

# 6. 安全评估和测试

```
SOC报告:
SOC 1 - 财务报告内部控制有关
SOC 2/3 - 设计和运行有效性报告
        - 包含安全性,可用性,机密性,处理完整性和隐私
SOC 3 - 用于大范围用户通报,而不需披露细节控制和测试结果

Type 1 - 在某个时间点有效
Type 2 - 在一段时间内有效

SOC 1 Type 1 - 证明内部财务控制设计合理
      Type 2 - 证明内部财务控制已经有效运行一段时间

SOC 2 Type 1 - 提供当下控制设计合理性证明
      Type 2 - 提供1年的观察确认控制的有效性

SOC 3 是 SOC 2 Type 2 报告的摘要版

常见漏洞:
不恰当的编程模式 - 缺少数据检查,SQL注入
基础设施 - 权限过大,脆弱的加密
设施功能错误 - 设施本身不限制访问
逻辑错误 - 用户下订单而不用支付

漏洞描述:
SCAP - CVE 通用漏洞披露, 漏洞唯一的标识符
     - CCE 通用配置枚举
     - CPE 通用平台枚举
     - CVSS 通用漏洞评分系统 - 0-10分,7分以上是严重漏洞
     - XCCDF 可扩展配置清单说明格式
     - OVAL 开放式漏洞和评估语言

渗透测试:
不只是发现漏洞,还要利用漏洞,作为安全评估
报告提交给管理层
要签署授权书,免死金牌
步骤 - 发现/收集, 枚举, 脆弱性勘探(人为,物理,系统和网络), 利用(未授权访问), 报告管理层和安全建议
黑盒 - 零知识,一无所知 - 真实模拟
白盒 - 全部知识,了解目标情况
灰盒 - 部分知识,不提供访问权限

双盲 - 都不知道
针对性测试 - 测试人员以普通用户提权

战争拨号 - 利用通过未经授权安装的调制解调器,拨入内部计算和网络资源访问

代码评审和测试:
动态测试 - 程序运行中的测试
静态测试 - 程序不需要运行, 源代码和编译后的程序,无法发现程序的逻辑错误

威胁建模 - 利用业务用例或使用场景,软件开发过程,潜在控制措施

开发阶段 - 静态代码分析和手动代码评审,检测不安全的编程,过时的代码库以及错误配置

模糊测试 - 发送更大块的随机数据,引起程序崩溃,利用缓冲区溢出

合成交易/主动检测 - 提升价值 7x24系统可用性监控, SLA

SAST 静态应用程序安全测试
DAST 动态应用程序安全测试
IAST 交互式应用程序安全测试
RASP 运行时应用程序安全保护

日志评审 - 审计和取证调查, 保护日志的完整性,机密性和可用性, 定期分析日志数据
         - 确保日志来源系统保持时钟同步

账号管理:
新员工签署可接受使用策略
仅在必须时才使用特权账号,日常维护使用受限账号
暂停账号 - 从人力资源部获取短期,长期离岗 离职人员名单

         - 评审输入是相关审计的结果,包括外部审计和内部审计

SwA 软件保障/软件担保 - 软件达到安全要求
